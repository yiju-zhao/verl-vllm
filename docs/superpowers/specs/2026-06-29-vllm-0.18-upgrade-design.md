# Design: Upgrade verl port vllm support 0.11 → 0.18 (CUDA + NPU)

- **Date:** 2026-06-29
- **Status:** Milestone A (CUDA) implemented & smoke-passing on vllm 0.18.1.dev0; Milestone B (NPU) code complete, smoke test pending remote Ascend access
- **Repo:** `drkernel-verl-port-drkernel/` (fork of verl; git baseline `b06d3ec`)

## 1. Goal

Make the verl port compatible with **vLLM 0.18** on **both** backends, with
*install + a rollout smoke test passing* as the definition of done on each:

- **CUDA** — on this machine (NVIDIA GB10, sm_121, aarch64, torch 2.9.1+cu130).
- **Ascend NPU** — via `vllm-ascend` 0.18, validated on a **remote Ascend machine**
  the user will provide access to.

Starting point: env `drkernel310` has vLLM **0.11.2** (CUDA wheel) installed; the
verl code carries version-gated branches up to ~0.16/0.17 but has **no 0.18 branch**
and pins `vllm>=0.8.5,<=0.12.0`.

## 2. Hard constraints (drive the whole plan)

1. **No NPU hardware here.** `torch_npu` is absent → NPU path is code-edit +
   source cross-check (against `vllm-ascend/` 0.18.0) + limited static import here;
   real install + smoke runs on the remote Ascend box.
2. **No stock CUDA wheel for sm_121/cu130 + 0.18.** torch 2.9.1 only officially
   covers ≤ sm_120; GB10 is sm_121. Expect to **build vLLM 0.18 from the local
   `vllm/` 0.18.0 source checkout**, possibly with build patches.
3. **No existing 0.18 reference anywhere.** Upstream verl `main` also tops out at
   0.16/0.17 and caps `vllm<=0.12.0`. 0.18 support is genuinely new and discovered
   empirically (install → run → fix → repeat).

## 3. Scope decisions (confirmed with user)

- Approach: **C (hybrid)** — targeted static API audit first, then empirical
  install/smoke loop as backstop.
- **`git init` the port** as a safety baseline — done (`b06d3ec`).
- **`vllm_omni`** (the separate omni/diffusion integration) is **out of scope** for
  the smoke test; we only avoid breaking its import.
- Smoke-test model: **Qwen3 small dense model — `Qwen3-0.6B`** (CUDA), minimal
  GRPO/PPO-style rollout; same minimal rollout on NPU.

## 4. Approach C — hybrid, in detail

### Phase 0 — Safety baseline  *(this machine)*
- `git init` + baseline commit. **(done)**
- Record the current (0.11.2) import inventory of verl's vllm integration modules
  as a regression reference (what must still import after the upgrade).

### Phase 1 — Shared verl code layer  *(static; affects both backends)*
- Version pins: `setup.py` `VLLM_REQUIRES` (`vllm>=0.8.5,<=0.12.0` → allow 0.18),
  `requirements*.txt`, and any ceiling logic in
  `verl/third_party/vllm/__init__.py` and `verl/workers/rollout/vllm_rollout/__init__.py`.
- Version-gated code that **both** backends hit: add/extend 0.18 branches where the
  gates currently top out — primarily `verl/utils/vllm/vllm_fp8_utils.py`
  (0.16+/0.17 branches) and `verl/utils/profiler/config.py` (0.13 gate). Each new
  branch is justified by a concrete diff against vLLM 0.18 source.

### Phase 2 — CUDA  *(this machine; full smoke test)*
- **Build/install vLLM 0.18** from local `vllm/` 0.18.0 source for sm_121/cu130
  against the existing torch 2.9.1. Capture any build patches needed for GB10.
- **Static API audit** of the CUDA-only surface verl imports (enumerated list of
  `from vllm…` symbols), focused on the fragile areas:
  - quantization internals: `model_executor.layers.quantization.utils.marlin_utils_fp4`,
    `…utils.fp8_utils`, `vllm._custom_ops`, `model_executor.parameter`;
  - model classes: `model_executor.models.{qwen3_moe,qwen3_next,qwen3_5,qwen2_moe,mixtral,kimi_vl,…}`;
  - engine/runtime: `vllm.LLM`, `vllm.outputs.RequestOutput`,
    `vllm.v1.engine.async_llm.AsyncLLM`, `vllm.v1.engine.FinishReason`,
    `vllm.distributed.parallel_state`, `distributed.utils.StatelessProcessGroup`.
  Patch confirmed breakages; leave the rest for the empirical loop.
- **Empirical loop:** import verl rollout modules → init engine → run the
  `Qwen3-0.6B` rollout smoke test; fix each failure (adding 0.18 gate branches as
  needed) until the smoke test passes. Debugging follows systematic-debugging.

### Phase 3 — NPU  *(remote Ascend; full smoke test)*
- **Cross-check `vllm-ascend` 0.18 source** for the private symbols
  `npu_vllm_patch.py` depends on: `SequenceRowParallelOp` (`ops.linear_op`),
  `MoECommType` / `select_moe_comm_method` (`ascend_forward_context`),
  `AscendDeviceType` / `get_ascend_device_type` (`utils`),
  `NPUModelRunner._select_moe_comm_method`, rotary-embedding `ApplyRotaryEmb`.
  Note which moved/renamed between the 0.13/0.14 branch and 0.18.
- **Add a 0.18 branch** to `npu_vllm_patch.py` (and the `is_A2` soc-version check,
  currently only `"0.11.0"`/`"0.13.0"`) plus any other NPU-gated code.
- On the **remote Ascend box**: install vLLM 0.18 + vllm-ascend 0.18, then run the
  minimal NPU rollout smoke test; iterate.

## 5. Validation strategy

| Backend | Static | Smoke test |
|---|---|---|
| CUDA | `import` of verl vllm-rollout modules under 0.18 (this machine) | `Qwen3-0.6B` minimal rollout (+ optional 1 GRPO/PPO step) — **this machine** |
| NPU | source cross-check + limited import | minimal rollout — **remote Ascend** |

Success = both smoke tests generate samples without error, and the Phase-0 import
inventory still imports.

## 6. Risks & mitigations

1. **sm_121/cu130 vLLM 0.18 build failure** (highest CUDA risk) → build from local
   source, capture patches; fall back to documenting the exact blocker if the GB10
   toolchain can't compile a needed kernel.
2. **Quantization internal churn** (fp8/fp4, `model_executor.parameter`,
   `_custom_ops`) → version-gated branches, validated against 0.18 source; the
   `Qwen3-0.6B` smoke path avoids fp8 so a quant break won't block the core DoD.
3. **vllm-ascend 0.18 private-API churn** → source cross-check before editing;
   NPU patch additions are gated strictly to 0.18.
4. **Remote Ascend access** is a Phase-3 prerequisite (user provides at that point).
5. **Empirical iteration count is unbounded** by nature → smoke test scoped to the
   smallest viable model to converge fast.

## 7. Out of scope

- `vllm_omni` / diffusion rollout (only keep its import non-broken).
- Performance tuning, multi-node, large models, fp8 training paths.
- Upstreaming changes back to verl-project/verl.

## 8. Definition of done

- verl port runs a `Qwen3-0.6B` rollout smoke test under vLLM 0.18 **on CUDA (this
  machine)** and a minimal rollout under vLLM 0.18 + vllm-ascend 0.18 **on the
  remote Ascend machine**, with version pins and version-gated code updated for 0.18
  and the baseline import inventory still importing.
